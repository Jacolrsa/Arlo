# Arlo - Project Decisions

This document records important architectural and development decisions.

The purpose is to explain **why** decisions were made, not just **what** was done.

Rules:

- Every significant design decision must be recorded here.
- Decisions are never deleted.
- If a decision changes, mark the old one as Superseded and add a new entry.
- Include the reason behind each decision.
- Include the version in which the decision was made.

---

# Decision 001

**Date**

2026-07-03

**Version**

v0.10

**Title**

Start Arlo v0.10 from a Clean Architecture

**Decision**

The prototype codebase will not continue evolving indefinitely.

Arlo v0.10 will become the first stable architecture.

Existing working ideas will be preserved, but incompatible experimental code will be removed.

**Reason**

The prototype accumulated several incompatible architectural changes that made further development increasingly difficult.

A clean foundation allows future versions to focus on features rather than infrastructure.

**Status**

Final

# Decision 002

**Date**

2026-07-03

**Version**

v0.10

**Title**

Generate Complete Files Only

**Decision**

Every code change generated by ChatGPT must be supplied as a complete file.

Partial edits are only allowed when explicitly requested.

**Reason**

Replacing complete files greatly reduces copy-and-paste mistakes and keeps the project consistent.

**Status**

Final

# Decision 005

**Date**

2026-07-03

**Version**

v0.10

**Title**

Freeze the Architecture after v0.10

**Decision**

After Arlo v0.10 is completed, the core architecture will be considered stable.

Future versions should add features rather than redesigning the framework.

Architectural changes should only occur when fixing significant defects.

**Reason**

Frequent redesigns slow development and increase maintenance effort.

**Status**

Final

---

# Decision 006

**Date**

2026-07-03

**Version**

v0.10

**Title**

Version-Based Development

**Decision**

Development will follow clearly defined versions.

Example:

- v0.10 Framework
- v0.11 MeshCore Monday
- v0.12 Leaderboards
- v0.13 Achievements

Each version must be stable before work begins on the next version.

**Reason**

Provides predictable development and simplifies testing.

**Status**

Final

**Decision**

# Decision 009

**Date**

2026-07-03

**Version**

v0.10

**Title**

One Responsibility Per Module

**Decision**

Each Python module should have one clear responsibility.

Examples:

listener.py
Receives MeshCore events.

router.py
Routes commands.

messenger.py
Handles outgoing message transmission.

storage.py
Handles persistent storage.

commands/
Contains user commands only.

**Reason**

Single-purpose modules are easier to understand, test, and maintain.

**Status**

Final

**Date**

**Version**

v0.10

**Title**

Record Future Ideas Instead of Interrupting Development

**Decision**
